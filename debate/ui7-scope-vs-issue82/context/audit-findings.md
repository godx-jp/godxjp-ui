# 70-agent naming/design audit — flagged components (47)

(Sonnet reviewers were aggressive; treat as candidates, not verdicts.)

- PageContainer: name=rename(->PageShell), rec=redesign-api(->Split into PageHeader (title + subtitle + extra + breadcrumb as ReactNode) + let children compose PageBody/PageFooter, matching Carbon PageHeader + Fluent PageHeader patterns.), sev=medium. The name "PageContainer" signals a width-clamping box (cf. MUI Container) rather than a header+body+footer shell; "PageShell" or keeping it as "PageHeader" (Carbon/Fluent) better communicates intent. More critically, embedding react-router-dom Link directly in a UI-library primitive creates a hard router dependency that breaks portability — Carbon and Fluent both accept pre-composed breadcrumb ReactNodes to stay framework-neutral.
- Inline: name=standard, rec=redesign-api(->Stack (if one exists as a column-direction sibling)), sev=medium. The name "Inline" is legitimate (Braid/SEEK uses it identically for a horizontal wrapping flex cluster), so no rename is needed. However, the component only surfaces `gap` and silently hard-codes `align-items: center` and `flex-wrap: wrap`, which violates composition-over-configuration — callers needing `align-items: flex-start` or `flex-wrap: nowrap` must bypass the component with raw `className`. The sibling `Stack` component (column direction) likely shares the same shape and both should be unified into a single `Flex`/`Stack` primitive with `direction`, `align`, `wrap`, and `gap` props, following the pattern used by Chakra HStack/VStack, Radix `Flex`, and Polaris `InlineStack`/`BlockStack`.
- ResponsiveGrid: name=rename(->SimpleGrid), rec=redesign-api(->SimpleGrid (rename + expand columns to number | ResponsiveValue)), sev=medium. Chakra UI's SimpleGrid and MUI's Grid both expose a columns (or xs/sm/md) prop directly on a meaningful primitive; wrapping a single div with a hard-coded enum and a data-attribute is a thin-wrapper anti-pattern that adds a named abstraction without adding real value. Rename to SimpleGrid (matching Chakra's canonical term) and replace the 2|3|4 enum with a flexible number or responsive object so consumers are not forced outside the component for anything outside those three values.
- AppShell: name=standard, rec=redesign-api(->Sidebar), sev=medium. The name "AppShell" is legitimate and matches Mantine's canonical term exactly, so no rename is needed. However, the prop API violates the composition-over-configuration principle by mixing a full-slot override (topbar) with partial sub-slots (topbarLeft/topbarRight/logo) — Ant Design's Layout keeps each region as a single slot and lets the caller compose the content inside it. The sidebarCollapsed prop should be removed from AppShell and kept solely inside the Sidebar component to restore single responsibility.
- Sidebar: name=standard, rec=redesign-api, sev=medium. The name "Sidebar" is correct and standard (used verbatim by shadcn/ui and Carbon's SideNav pattern). The design issue is the data-driven, configuration-object API: passing `sections: SidebarSectionProp[]` forces the entire tree through a single opaque prop and prevents per-item customisation without an API change, which is the anti-pattern Carbon, shadcn/ui, and Fluent UI all explicitly avoid by shipping composable sub-components (SidebarItem, SidebarGroup, SidebarSection). The product-branding header slot further violates single responsibility — it belongs in an AppShell or a dedicated Header primitive.
- Topbar: name=rename(->AppBar), rec=redesign-api(->AppBar (rename) with slot-based composition API (left/center/right slots or children) and no domain-specific props), sev=medium. MUI's AppBar+Toolbar and Carbon's Header demonstrate the correct pattern: a domain-neutral horizontal container that composes arbitrary children/slots, leaving product-switcher or search widgets to be composed in by the consuming app. Baking product, project, unread, and onTweaksOpen into the primitive couples the design system to one application's domain model, making it non-reusable; the name "Topbar" is also non-canonical versus the mainstream "AppBar" (MUI, Chakra) or "Header" (Carbon, Ant, Fluent).
- PageInset: name=rename(->PageGutter), rec=merge(->PageContainer), sev=medium. PageInset is a single-`<div>` wrapper that only re-applies the page horizontal gutter token, which is already owned by `PageContainer`. No canonical design system (MUI, Ant, Carbon, Fluent) ships a standalone component for this; they model it as a prop or sub-region of the container itself (e.g. MUI `Container` with `disableGutters`, Carbon `Grid` padding props). Additionally, "inset" is a false friend — in CSS and most design systems it denotes inner shadow or offset positioning, not horizontal padding re-alignment, making the name actively misleading to international contributors.
- SplitPane: name=rename(->SidebarLayout or TwoColumnLayout), rec=rename(->SidebarLayout), sev=medium. "SplitPane" is universally understood as a resizable/draggable split container (react-split-pane, allotment, Fluent UI SplitView), so naming a static two-column layout "SplitPane" creates a misleading affordance for every consumer. Renaming to "SidebarLayout" (or "TwoColumnLayout") aligns the name with its actual behaviour — a fixed main + aside shell — matching the pattern used in Ant Design Layout+Sider and Carbon grid-with-side-panel documentation.
- Breadcrumb: name=standard, rec=redesign-api, sev=medium. The name is universally standard, but the API violates the composition-over-configuration principle endorsed by every major design system: shadcn/ui ships BreadcrumbItem + BreadcrumbLink + BreadcrumbPage + BreadcrumbSeparator as discrete sub-components, Radix and Chakra follow the same compound pattern, and MUI exposes a `separator` prop with individually-rendered children. A single `items` prop also hard-couples the component to react-router-dom's Link, breaking portability outside that router context.
- DataTable: name=standard, rec=redesign-api, sev=medium. The name "DataTable" is canonical and matches Carbon, MUI X, and AG Grid. However, the DataTable.Pagination sub-component encodes cursor pagination as the only strategy (cursor/hasMore), a business-layer assumption that domain-neutral UI libraries like MUI X and Carbon avoid by accepting a generic render slot or a pagination-mode prop. Internally, raw &lt;input type="checkbox"&gt; is used instead of the library's Checkbox primitive, breaking the composition contract the library's own CLAUDE.md mandates.
- Card: name=standard, rec=redesign-api(->Extract CardStat into a standalone StatCard (or Stat) component in a separate file; collapse CardDensity into the existing `size` prop or CSS tokens so the Card API has a single spacing axis), sev=medium. Every major design system (shadcn/ui, MUI, Ant Design, Chakra) keeps Card as a pure surface container with no domain-flavored sub-components — KPI stat tiles are always separate composites. Shipping CardStat inside the Card primitive couples dashboard business logic to the base container, and the dual spacing axes (size + density) duplicate responsibility within the same prop API.
- Badge: name=standard, rec=keep, sev=low. The name "Badge" is internationally standard (shadcn/ui, Chakra, Fluent UI, MUI all use it for inline label/status chips). The component is a correct, lean primitive — CVA variants, HTMLAttributes passthrough, no domain logic — and matches shadcn/ui's design exactly. The only gap is the missing role attribute on the root div: when variant conveys status (success/warning/destructive), screen readers get no signal, which Fluent UI and MUI address with role="status".
- Timeline: name=standard, rec=redesign-api, sev=medium. The name "Timeline" is the cross-system standard (Ant Design, MUI @mui/lab, Chakra UI v3, Carbon all ship it under that name), so no rename is needed. However the current implementation violates domain-neutrality and composition-over-configuration: it hardcodes travel icons (Plane, CheckCircle2) and a domain-specific `location` field that have no place in a generic UI primitive, and the flat `items[]` array API prevents per-item slot customisation that all major systems (Ant Design's Timeline.Item, MUI's TimelineItem + TimelineContent slots, Chakra's compound Timeline.Item/Timeline.Connector) provide through a compound/slot model.
- DataState: name=rename(->QueryBoundary), rec=redesign-api(->QueryBoundary (or defer to React Suspense + ErrorBoundary at the app layer)), sev=medium. The name "DataState" is not used by any mainstream design system for this pattern; the community-standard name for a TanStack Query lifecycle boundary is "QueryBoundary" (react-query-kit, Suspensive) or "AsyncBoundary" (Suspensive @suspensive/react). The hard coupling to `UseQueryResult` violates the principle that a UI library should be data-fetching-agnostic — Ant Design, MUI, and Carbon ship no such component precisely because it belongs in the data layer, not the component library layer; if kept, the prop type should accept a duck-typed interface (`{ isPending, isError, isFetching, error, data, refetch }`) rather than a concrete TanStack import.
- InfiniteQueryState: name=rename(->InfiniteList), rec=redesign-api(->DataState), sev=medium. The "State" suffix and "Query" infix name the TanStack Query implementation detail rather than the rendered affordance (a paginated list with load-more footer), making the name non-portable and non-standard across every major design system. The flatten prop conflates data-shape normalisation with rendering, a single-responsibility violation; Ant Design's List with loadMore and MUI's infinite-scroll patterns keep the page-flattening in the data layer, passing a flat array to the view component.
- MutationFeedback: name=rename(->MutationError), rec=merge(->AlertQueryError), sev=medium. No mainstream design system (MUI, Ant, Chakra, Fluent, Carbon, shadcn) names a component after a data-fetching library concept like "mutation" — the name leaks TanStack Query vocabulary into the public API. Architecturally, the component is a near-pass-through wrapper over AlertQueryError with a trivial isPending/isError gate; MUI and Ant Design handle this pattern via props on their Alert primitive rather than a separate named component, which is the correct abstraction level.
- FormField: name=rename(->Field), rec=rename(->Field), sev=low. The component is a well-designed, single-responsibility layout wrapper that wires together a Label, an arbitrary control (via children + React.cloneElement for a11y injection), a helper text, and an error message — exactly what Chakra UI calls "FormControl", Fluent UI calls "Field", Ant Design calls "Form.Item", and Carbon calls "FormGroup". The name "FormField" is reasonable but slightly over-qualified: Fluent UI 9, Radix Themes, and react-aria all converged on the shorter "Field" (e.g. TextField = Field wrapping a text input), which signals that the primitive is useful outside a `<form>` too. The design itself is correct — single responsibility, composition via children, a11y attributes injected via cloneElement rather than baked into each control, no business logic, semantic tokens used for error/muted colours.
- SearchInput: name=standard, rec=redesign-api(->Input), sev=medium. The name "SearchInput" is used by Carbon (Search), Ant Design (Input.Search), Fluent UI (SearchBox), and React Aria (SearchField), so the concept is legitimate as a standalone primitive. However, the current implementation violates single-responsibility by embedding debounce and mixed controlled/uncontrolled state internally — Carbon and React Aria's SearchField expose a clean value/onChange API and leave debounce to the consumer, which is the correct abstraction boundary for a domain-neutral UI primitive.
- Select: name=standard, rec=redesign-api(->SearchSelect), sev=medium. Merging two abstraction levels behind a single discriminated-union root violates the single-responsibility principle and produces an opaque, hard-to-type API: callers cannot tell which contract applies without reading the implementation, and TypeScript must accept a union that conflates compound-primitive props with data-driven props. The correct pattern (used by Radix, shadcn/ui, and MUI) is to keep the compound primitive (Select + SelectTrigger + SelectContent + SelectItem) as the foundational API and offer the data-driven convenience as a clearly named higher-level component (e.g. SelectField or keep SearchSelect un-deprecated), rather than folding both into one overloaded root.
- Checkbox: name=standard, rec=redesign-api(->CheckboxGroup.onChange → rename to onValueChange with signature (value: string[]) to mirror RadioGroup and Radix shared vocabulary), sev=low. The name "Checkbox" is canonical across every major design system (Radix, MUI, Ant Design, shadcn/ui, Carbon, Fluent, Chakra, React Aria). The component itself is a correct, minimal Radix primitive wrapper with a well-structured Group sub-component via Object.assign; the only issue is the prop naming drift between CheckboxGroup (onChange) and RadioGroup (onValueChange), which violates the shared vocabulary / concept-first prop API principle cited by Radix and MUI.
- DatePicker: name=standard, rec=redesign-api(->DatePicker), sev=low. The name is universally standard across MUI, Ant Design, Fluent UI, Carbon, and react-aria. The component's abstraction level, single responsibility (ISO-8601 string value, typeable input + calendar popover), a11y (WAI-ARIA combobox role, aria-expanded, aria-haspopup), and composition are all correct. The only issues are prop-vocabulary inconsistencies: `onChange` should be `onValueChange` to align with the shared godx-ui vocabulary used by Select/RadioGroup/Switch, and the `locale` prop leaks the internal react-day-picker type rather than accepting a plain BCP-47 locale string — a minor API seam improvement.
- Dialog: name=standard, rec=redesign-api(->AlertDialog (split out); DialogConfirm (move to a patterns/presets layer)), sev=medium. The `mode="confirm"` branch silently changes ARIA role, keyboard behavior (alertdialog blocks Escape), and available sub-components — a hidden behavioral fork that violates single responsibility and surprises consumers who rely on consistent semantics. Radix UI, shadcn/ui, Chakra UI, and react-aria all keep Dialog and AlertDialog as separate primitives precisely because their ARIA roles and interaction contracts differ; merging them under one `mode` prop is an anti-pattern that should be split into `Dialog` (role=dialog) and `AlertDialog` (role=alertdialog), with `DialogConfirm` promoted to a higher-level pattern/preset layer.
- SkeletonTable: name=standard, rec=merge(->DataTable (as its built-in loading/skeleton state via the existing `loading` prop)), sev=medium. SkeletonTable duplicates layout logic already owned by DataTable's `loading` prop, violating single-responsibility and creating a drift risk when DataTable's column padding or row height changes. Carbon's DataTableSkeleton is the canonical precedent, and Carbon ships it as a tightly-coupled companion to DataTable — the right fix is to ensure DataTable's `loading` prop fully covers the use case and deprecate the standalone export.
- SkeletonCard: name=rename(->SkeletonStat), rec=rename(->SkeletonStat), sev=medium. The component is a 3-block skeleton (label + large value + subtext) that mirrors a stat/KPI tile — the layout that Ant Design calls "Statistic", MUI X calls "Stat", and Carbon calls "Metric tile". The name "SkeletonCard" collides with the generic Card container, confuses developers looking for a card-shaped skeleton vs a stat-shaped one, and breaks the library's own naming convention (SkeletonTable, SkeletonRows, SkeletonDetail all discriminate by content shape, not wrapper). Renaming to SkeletonStat aligns with both mainstream systems and the existing family convention.
- FilterBar: name=rename(->FilterToolbar), rec=redesign-api(->Toolbar (layout primitive) + caller-controlled clear Button), sev=medium. No mainstream design system (MUI, Carbon, Fluent UI, Ant Design, shadcn/ui) ships a standalone "FilterBar" primitive — the pattern is always a Toolbar/CommandBar composition where filter controls are children and a clear action is a plain Button. The current implementation violates single-responsibility by coupling layout, filter-state awareness (hasActiveFilters), and i18n-translated UI copy inside a generic primitive, making it untestable in isolation and impossible to adapt to non-filter toolbar use cases.
- FilterGroup: name=rename(->FilterSection), rec=redesign-api(->FilterBar), sev=medium. The component name embeds a product concept ("Filter") into a purely structural layout primitive; MUI, Carbon, and react-aria all call this pattern FormGroup or Group and attach role="group" with aria-labelledby for screen-reader support, which this component omits entirely. The fix is to rename it to something like FilterSection (scoped to its FilterBar sibling) and add the correct ARIA group semantics.
- Steps: name=standard, rec=redesign-api, sev=medium. The name "Steps" is internationally standard (Ant Design, Chakra UI both use it verbatim), but the flat items-array API breaks composition-over-configuration — MUI Stepper, Chakra v3 Steps, and Radix all expose compound <Step>/<StepLabel> children that give per-step render flexibility the current config object cannot express. The onChange/onValueChange mismatch and hardcoded aria-label further diverge from shared design-system vocabulary.
- Cascader: name=rename(->TreeSelect), rec=rename(->TreeSelect), sev=medium. "Cascader" is Ant Design-proprietary jargon; the broader ecosystem (MUI community, rc-tree-select, HeadlessUI patterns) calls this concept "TreeSelect", making it the more internationally recognizable name. The component's logic and composition are sound (Popover + columns + Checkbox + Command search), but the ARIA contract must be corrected — role="combobox" on the trigger requires a paired listbox or tree role on the popup with proper aria-controls/aria-activedescendant, as specified by the ARIA 1.2 combobox pattern.
- TreeSelect: name=standard, rec=redesign-api(->TreeSelect), sev=medium. The name "TreeSelect" is the canonical term used by Ant Design and MUI X for this exact control (a dropdown trigger that reveals a hierarchical/tree-structured option list), so the name is correct and internationally recognised. However the API conflates three distinct behaviours (single, multi, checkable) via boolean flags rather than a variant prop or separate components, mirrors Ant Design's domain-specific showCheckedStrategy enum as static constants on the export, mixes role="combobox" on the trigger with role="tree" in the panel (ARIA requires combobox → listbox, not combobox → tree), and hard-codes the empty state instead of accepting an empty slot — all of which violate the single-responsibility and composition-over-configuration principles expected of a design-system primitive.
- Upload: name=standard, rec=redesign-api(->Split into Upload (dropzone + button + picture-card variants only) and a separate AvatarUpload (or ImageUpload) component that owns the draft/crop/undo lifecycle — matching Ant Design's separation where Upload handles generic file input and avatar/profile photo is composed independently.), sev=medium. The name "Upload" is canonical (Ant Design Upload, Carbon FileUploader, Fluent UI FileUpload all use it), but the component violates single responsibility by bundling avatar-specific crop, draft state, and undo/replace semantics alongside a plain dropzone/button — Ant Design keeps these concerns separate, and the avatar variants' use of useUploadDraft and UploadCropDialog is dead weight for every non-avatar consumer.
- UploadCropDialog: name=rename(->ImageCropDialog), rec=redesign-api(->ImageCropDialog), sev=medium. The "Upload" prefix encodes caller context into the component name rather than describing its own responsibility — all comparable systems (Ant Design ImgCrop, react-easy-crop Cropper) name the component after the crop action, not the upload flow it participates in. More critically, avatar-specific constants (256 px, circle shape, JPEG 0.92) are hardwired, making this a product widget rather than a reusable primitive; the fix is to expose aspectRatio, outputSize, cropShape, and outputFormat as props with sensible defaults.
- ColorPicker: name=standard, rec=redesign-api, sev=low. The name is universally canonical (Ant Design, Fluent UI, Carbon all ship "ColorPicker"). The implementation is a well-scoped primitive with correct a11y, but `onChange(hex: string)` deviates from the shared `onValueChange` vocabulary that every other controlled input in the library uses — Ant Design and MUI both standardize on a single value-change callback signature per input type, and godx-ui's own Input/Select/DatePicker follow `onValueChange`. This vocabulary inconsistency breaks discoverability and makes the API harder to learn as a system.
- Slider: name=standard, rec=redesign-api, sev=low. The name is universally canonical (Radix, MUI, Ant, Chakra, Fluent, Carbon, shadcn all say "Slider") and the overall wrapper pattern is correct. The one issue is that the internal values memo overrides Radix's own [0] default with [min, max] when no value/defaultValue is supplied, making an uncontrolled single-thumb slider silently render as a two-thumb range slider — a behavioral surprise that no major design system replicates. The fix is to remove the fallback (let Radix handle its own default) and derive thumb count only from actual value/defaultValue when provided.
- Calendar: name=standard, rec=merge(->DatePicker), sev=medium. Every major system (shadcn/ui, MUI, Ant, Fluent) exposes Calendar as a standalone date-grid primitive with its own value/onValueChange API; this component delegates 100% of its contract to DayPickerProps, making it a styling shim rather than a true primitive. It should either own a normalised prop API (value, onValueChange, mode, disabled, locale) or be collapsed into DatePicker's popover panel, as Radix and react-aria do.
- CountrySelect: name=standard, rec=remove(->Select (compose Select + CountryOptionLabel at the app layer)), sev=medium. No mainstream design system (Radix, MUI, Ant, shadcn/ui) ships a CountrySelect primitive because country-domain logic belongs at the application layer, not in a UI framework. The component additionally breaks the standard controlled-input contract — missing value/onValueChange — that every peer primitive in the library follows, making it an inconsistent and leaky abstraction over the Select it wraps.
- ChoiceField: name=rename(->Field), rec=redesign-api(->Label (absorb description slot) or a fuller Field primitive modeled on Fluent UI Field), sev=medium. No mainstream design system (Radix, MUI, Ant, Fluent, Carbon, shadcn/ui, react-aria) ships a public "ChoiceField" abstraction — the canonical name for this layout unit is "Field" (Fluent UI) or "FormItem" (shadcn/ui). Beyond naming, the component is a thin wrapper that adds no logic over raw Label+p composition, and its omission of aria-describedby means the optional description is inaccessible to assistive technology, violating WCAG 1.3.1.
- Command: name=rename(->FilterableList (or keep as an internal, unexported implementation detail)), rec=merge(->SearchSelect / Cascader / TreeSelect (make it a private implementation detail, not a public export)), sev=medium. The "Command" name is borrowed directly from shadcn/ui's cmdk wrapper, where it is intended for ⌘K command-palette UIs; none of MUI, Ant Design, Chakra, Fluent, Carbon, or react-aria use "Command" to mean a filterable listbox. Because it has zero logic beyond CSS class injection and is only consumed internally by SearchSelect, Cascader, and TreeSelect, it should either be unexported (private implementation detail) or renamed to something that reflects its role (e.g., FilterableList), preventing consumers from depending on a cmdk-specific surface that the library may want to swap out.
- CheckboxGroup: name=standard, rec=redesign-api(->RadioGroup (align to its onValueChange convention)), sev=medium. The name "CheckboxGroup" is canonical — Chakra UI, React Aria, Ant Design (Checkbox.Group), and Carbon all use this exact name. The design issue is the divergent event prop: RadioGroup exposes `onValueChange` (Radix convention) while CheckboxGroup exposes `onChange`, breaking internal API consistency; all choice-group siblings should use `onValueChange` (or at minimum the same name) so consumers learn one vocabulary across the whole data-entry surface.
- Radio: name=standard, rec=redesign-api(->RadioGroup (make the group the primary export; demote RadioItem to Radio.Item sub-export only)), sev=medium. The name "Radio" with a `.Group` namespace follows Ant Design's pattern and is internationally recognized, so the name is acceptable. However, assigning the compound object to the bare `RadioItem` function means `<Radio />` renders a context-less indicator dot — a design mistake that contradicts every peer system (Radix, MUI, shadcn/ui, react-aria all make the group the primary callable). The primary export should be `RadioGroup` (or `Radio.Group` as the callable root), with `Radio.Item` as the subordinate primitive.
- Collapsible: name=standard, rec=redesign-api(->@radix-ui/react-collapsible (direct) or merge styling/defaults here to justify the wrapper), sev=medium. The name "Collapsible" is canonical and shared by Radix UI, shadcn/ui, and React Aria (as Disclosure). The design problem is that the component is a zero-value re-export: no className wiring, no default animation, no token application — nothing shadcn/ui adds when it does the same pattern. Either delete the file and import Radix directly, or give it a real implementation (animated content, token-based styling via cn/cva) to justify the abstraction layer.
- TreeList: name=rename(->Tree (or TreeView)), rec=redesign-api(->Tree), sev=high. Every major design system (Ant Design Tree, MUI TreeView, Fluent UI Tree, Carbon TreeView, react-aria Tree) names this pattern "Tree" or "TreeView" — "TreeList" is non-standard. More critically the implementation is broken as a primitive: the hardcoded Package icon leaks domain concerns, there are zero ARIA tree roles making it a11y-incorrect, and the flat-array-with-depth data model diverges from every canonical tree API; it needs a full redesign with recursive node data, proper aria attributes, and an icon slot prop before it can ship as a design-system component.
- LocalePicker: name=rename(->LocaleSelect), rec=redesign-api(->Select), sev=medium. The component hardcodes APP_LOCALES (["vi","en","ja"]) inside a design-system primitive, making it non-reusable outside this specific product's locale set — design systems like MUI and Ant Design never embed application data in primitives. The correct pattern is either a bare Select (consumer passes locale options) or a generic LocaleSelect that accepts a `locales` prop; the AppProvider coupling should be opt-in via a separate hook, not a throw-on-missing-context inside the primitive itself.
- TimezonePicker: name=standard, rec=redesign-api(->LocalePicker, DateFormatPicker, TimeFormatPicker), sev=low. The name "TimezonePicker" is correct and standard (matches react-timezone-select, MUI community, and the library's own -Picker family). The design concern is that TimezonePicker, LocalePicker, DateFormatPicker, and TimeFormatPicker are structurally identical Select wrappers — same prop shape, same AppProvider fallback pattern, same throw guard — violating DRY at the component-factory level; Ant Design and MUI solve this with a single configurable Select primitive rather than per-domain wrapper components. The hardcoded sm:w-56 width also violates composability (MUI and Radix leave sizing to the consumer via className/sx).
- DateFormatPicker: name=rename(->DateFormatSelect), rec=merge(->Select (pass options and value externally; move DateFormatPicker to an app-settings layer or @godxjp/ui/app package alongside LocalePicker/TimezonePicker)), sev=medium. The component violates domain-neutrality — a core design-system primitive must not encode an app-specific enum (AppDateFormat) or depend on a product AppProvider. In every major system (MUI, Ant, shadcn/ui, Carbon) this pattern is handled by composing a generic Select with caller-supplied options, not by publishing a named component per option-set type. The name "Picker" further conflicts with the industry-standard meaning established by MUI DatePicker, Ant DatePicker, and react-aria DatePicker (all of which mean calendar-based date-value selection).
- TimeFormatPicker: name=rename(->TimeFormatSwitcher), rec=rename(->TimeFormatSwitcher (or merge as a prop into a LocalePreferences / AppPreferencesBar composite)), sev=low. The "Picker" suffix is canonically reserved for date/time value selection components (MUI TimePicker, Ant TimePicker, react-aria TimeField); using it for a 2-option display-format preference toggle causes naming collision with that well-established convention. Renaming to "TimeFormatSwitcher" aligns with the "Switcher" pattern (Carbon ContentSwitcher, locale/theme switchers) and correctly signals this is a settings/navigation widget, not a data-entry primitive.
- PrefetchLink: name=standard, rec=merge(->Link (or extract as a usePrefetchLink hook consumed by Link's onMouseEnter/onFocus)), sev=low. The name "PrefetchLink" is descriptive and unambiguous, matching TanStack Router's own vocabulary for the same concept. However, the component is a thin event-handler wrapper over react-router-dom's `<Link>` — a pattern TanStack Router itself eliminates by making preloading a first-class `preload` prop on `<Link>`. Splitting this into its own exported component fragments the link API and leaks query-layer concerns (queryKey, queryFn, staleTime) into the component surface; it is better expressed as a `usePrefetchLink` hook or merged into a `Link` wrapper that accepts an optional `prefetch` prop.
- QueryRefetchButton: name=rename(->RefetchButton), rec=merge(->Button), sev=medium. The component is a two-line wrapper that sets onClick and disabled on Button — both Ant Design and MUI solve this with a loading prop on their Button primitive rather than a separate component. The "Query" prefix ties a UI primitive to a specific data-fetching library (TanStack Query), which no mainstream design system (Radix, MUI, Chakra, Carbon) does; if kept as a standalone convenience helper it should live in an app-layer utilities file, not the core design-system component library.
