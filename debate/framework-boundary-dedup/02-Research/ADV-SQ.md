# ADV-SQ Research — status-quo, dedup-only

Assumptions:
- "Core" means one importable UI framework package, not necessarily one undifferentiated entrypoint; `@godxjp/ui/query`, `@godxjp/ui/admin`, and similar sub-exports can keep surfaces organized without package extraction.
- Judge component nature, not current app usage. Hardcoded labels/statuses are API polish issues unless the component itself is a business workflow.
- Peer-DS citations are real shipping facts: MUI ships AppBar, Container, Badge, Card, Skeleton, Date Pickers/Data Grid packages; Mantine ships AppShell, Flex, Stack, SimpleGrid, DatePicker; Ant Design ships Layout/Sider/Header, Cascader, TreeSelect, Calendar, DatePicker, Statistic, Badge, Skeleton, Upload, ConfigProvider; Chakra ships Stack/HStack/SimpleGrid/Card/Badge/Skeleton/Accordion; Fluent ships App surfaces such as AppItem/Nav/Toolbar/Field/Badge/Skeleton; Carbon ships Header/SideNav, Grid, DataTable, Tag/Badge-like status labels, DatePicker, Skeleton states.

Bottom line: extraction gives aesthetic purity but worsens consumer ergonomics and migration cost. The package should keep its broad kit and deduplicate only strict same-concept overlaps: `Stack/Inline -> Flex/Stack`, `Autocomplete -> SearchSelect`, `Cascader -> TreeSelect mode`, `Skeleton* -> Skeleton family`, `MutationFeedback/QueryRefetchButton` into query feedback/button helpers, `StatusBadge/CardStat` into `Badge/StatCard`, and `PageInset` into page layout.
